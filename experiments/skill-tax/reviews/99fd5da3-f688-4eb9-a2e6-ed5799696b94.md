## Setup: environment, imports, raw-body capture

Webhook secrets are per-endpoint and must come from the environment, not source. Required env vars per provider:

```
STRIPE_WEBHOOK_SECRET=whsec_...      # Stripe-issued, per-endpoint
GITHUB_WEBHOOK_SECRET=...            # operator-chosen, set in repo webhook settings
SLACK_SIGNING_SECRET=...             # Slack signing secret from app config
```

The receiver imports use Python's stdlib `hmac` for the constant-time comparison, `hashlib` for the digest algorithm, and FastAPI's `Request` for the raw body. Stripe's Python SDK provides a verification helper that wraps both:

```python
import hmac
import hashlib
import os
from fastapi import FastAPI, Request, HTTPException
import stripe                          # stripe>=10.0 — uses hmac.compare_digest internally
```

Middleware order matters. FastAPI parses JSON bodies via dependency injection at the route level, not via global middleware, so the raw bytes are available via `await request.body()` inside the route handler — but only if no upstream middleware has consumed the stream. The supported pattern (FastAPI docs, "Custom Request and APIRoute class", verified 2026-05-01) is to override `APIRoute.get_route_handler` so the raw body is read once and cached on the request before any body-parsing dependency runs. Alternative: read `await request.body()` inside the route function and parse JSON manually — simpler but couples verification to handler.
```

## Verifying the signature: HMAC-SHA256 with constant-time comparison

Every supported scheme uses HMAC-SHA256 over a sender-defined string-to-sign. The receiver computes the same HMAC with the shared secret and compares against the header. Comparison must use `hmac.compare_digest` (Python stdlib, verified 2026-05-01 against docs.python.org/3/library/hmac.html), never `==` — string equality short-circuits on the first differing byte and leaks the signature one byte at a time through timing side channels. `hmac.compare_digest` is constant-time for equal-length inputs and accepts either `bytes` or `str` for both arguments, but the two arguments must be the same type.

Python's `hmac.new(key, msg, digestmod)` requires `digestmod` since Python 3.8 (verified 2026-05-01). Pass it as a keyword argument: `hmac.new(secret, body, digestmod=hashlib.sha256)`. The fast path is `hmac.digest(key, msg, digest)` (added 3.7) which uses the optimized C implementation when `digest` is a string name like `"sha256"` and OpenSSL is available.

Two things to keep straight: the secret must be `bytes` (encode with `secret.encode("utf-8")` if you store it as a string), and the message hashed must be the raw request body bytes — not a re-serialized JSON object. Compute the expected digest with `hmac.new(secret_bytes, raw_body_bytes, digestmod=hashlib.sha256).hexdigest()` and pass both the computed and header-supplied hex strings to `hmac.compare_digest` directly.
```

## Per-vendor signature schemes

Different providers compose the string-to-sign differently. Get this wrong and verification silently fails for one provider while passing for another.

Stripe (verified 2026-05-01 against docs.stripe.com/webhooks/signatures): header `Stripe-Signature: t=<unix-ts>,v1=<hex-hmac-sha256>[,v0=<legacy>]`. The signed string is `<unix-ts>.<raw-body>`. Default replay tolerance is 300 seconds (5 minutes) when using the Python SDK helper `stripe.Webhook.construct_event(payload, sig_header, secret, tolerance=300)`. The helper raises `stripe.error.SignatureVerificationError` on bad signature OR stale timestamp; both should map to HTTP 400.

GitHub (verified 2026-05-01 against docs.github.com/en/webhooks/using-webhooks/validating-webhook-deliveries): header `X-Hub-Signature-256: sha256=<hex-hmac-sha256>`. The signed string is the raw request body, with no timestamp prefix. There is no header timestamp at all — replay protection falls entirely to event-id idempotency using the `X-GitHub-Delivery` UUID header. The `sha256=` prefix is part of the header value; strip or include it consistently when comparing.

Slack (verified 2026-05-01 against api.slack.com/authentication/verifying-requests-from-slack): headers `X-Slack-Signature: v0=<hex-hmac-sha256>` and `X-Slack-Request-Timestamp: <unix-ts>`. The signed string is `v0:<unix-ts>:<raw-body>`. Slack documents a 5-minute replay window — reject anything outside.

Standard Webhooks v1.0 (verified 2026-05-01 against standardwebhooks.com): headers `webhook-id`, `webhook-timestamp`, `webhook-signature: v1,<base64-hmac-sha256>`. The signed string is `<webhook-id>.<webhook-timestamp>.<raw-body>`. Signature is base64, not hex — adapt your decode call accordingly. Adopted by Svix, Zapier, Twilio, and others.
```

## Guardrails: never do these

These are the lines you do not cross when implementing a webhook receiver. Each item below has bitten production systems; the verification and example sections show the right shape.

- Never compare signatures with `==` or `!=`. String equality short-circuits on the first differing byte and leaks the signature one byte at a time through timing side channels. Use `hmac.compare_digest`, which is constant-time for equal-length inputs.
- Never call `await request.json()` (or any body-parsing dependency) before verifying the signature. Once the body is parsed and re-serialized, the bytes you hash will not match the bytes the sender hashed. Verification will silently fail. Read raw bytes via `await request.body()` first, verify, then parse JSON yourself.
- Never hard-code the webhook secret. Secrets in source rotate by force-push and leak through git history. The receiver must read from env or a secret manager. `grep -rn 'whsec_' src/` should return nothing.
- Never trust an unsigned event. No fallback path that skips verification "for testing" or "when the header is missing" — those paths ship to production. Reject with 400 or 401, no exception.
- Never use HMAC-SHA1 (or anything else) for new receivers. GitHub's legacy `X-Hub-Signature` header is SHA-1 and is deprecated; new receivers use `X-Hub-Signature-256`. Do not accept SHA-1 even as a fallback.
- Never call `hmac.new(key, msg)` without `digestmod`. Python 3.8 made `digestmod` required (verified 2026-05-01); the two-argument form raises `TypeError` on 3.8+ and was silently insecure on older runtimes.
- Never leave the dedup row after the handler fails. A persisted row plus a 5xx response means the sender's retry hits the unique constraint, gets a 200, and the event is silently lost. Either roll the row back on failure or wrap (insert + handler work) in one transaction.
- Never return 4xx on duplicate delivery. A 4xx tells the sender to keep retrying; a 200 tells it to stop. Duplicates are a successful no-op from the sender's perspective.
- Never trust the timestamp without checking tolerance. A signed request whose timestamp is unbounded is a forever-valid replay token. Reject anything older than 300 seconds (or the sender's documented tolerance).
- Never log raw signatures, secrets, or full request headers at info level. They become a credential exfiltration surface. Redact in logs; sample-debug only in dev.

import hmac
import hashlib
from fastapi import FastAPI, Request

app = FastAPI()

@app.post("/webhook/stripe")
async def handle_stripe_webhook(request: Request):
    # Read the raw body first to ensure we have access to it for signature verification
    payload = await request.body()
    
    # Get the signature from the header
    sig_header = request.headers.get("Stripe-Signature")
    if not sig_header:

    # Get the webhook secret from environment

    # Verify the signature using timing-safe comparison
    if not hmac.compare_digest(sig_header, expected_sig):
        return {"error": "Invalid signature"}, 400
