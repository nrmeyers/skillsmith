## Why this skill exists

JWT validation patterns center on three procedures: algorithm whitelisting on the receiver (never trust the token's `alg` header), full claim validation (`exp`, `iss`, `aud`, `nbf`), and disciplined key management (sufficient HS256 secret entropy, JWKS rotation for RS256). Skip any of the three and the signature check becomes security theatre — the token validates but the receiver still accepts forged or replayed tokens.

This skill is PyJWT-flavored (PyJWT 2.x) because the Python ecosystem's reference library exposes algorithm enforcement, claim validation, and exception taxonomy as first-class arguments — there is no "right" way to use it without those arguments set, and most JWT vulnerabilities in the wild come from receivers that omit them.

Use this skill when accepting JWTs from any source (frontend SPAs, mobile apps, B2B partners, internal services); reviewing existing JWT validation code for the algorithm-confusion family of attacks (RFC 8725); or implementing a new authentication middleware that must reject expired, foreign-issued, or wrong-audience tokens.

The skill covers receiver-side validation only. JWT issuance (key generation, signing, claim selection) is a separate concern handled by the identity provider; pair this skill with the identity provider's documentation when the same service does both.
```

### `jwt-validation-patterns:8` (`guardrail`)  matched: `scope`

```
## Guardrails: never do these

These are the lines you do not cross when validating JWTs. Each item below has bitten production systems; the execution and example sections show the right shape.

- Never call `jwt.decode(token, key)` without `algorithms=[...]`. PyJWT 2.x requires it; older code that omitted it accepted whatever the token's `alg` header claimed — including `none` or an attacker-chosen algorithm. RFC 8725 §3.1 makes algorithm whitelisting normative.
- Never include `none` in the algorithm whitelist. The `none` algorithm produces a token with no signature; accepting it accepts any forged token. There is no legitimate use case for `none` on a receiver.
- Never accept HS256 and RS256 simultaneously with the same configured key. The classic algorithm-confusion attack signs an HS256 token using the issuer's RS256 public key as the HMAC secret; if the receiver tries HS256 with that public key as the secret, it validates. Per scheme, one whitelist with one matching key.
- Never trust the `kid` header to resolve to a local file path or arbitrary URL. `kid` is an opaque identifier scoped to the issuer's JWKS; treat it as a lookup key, never as a filesystem or network resource. `PyJWKClient` does this correctly.
- Never skip claim validation. `exp` is non-optional. `iss` and `aud` must match server expectations. Tokens lacking `exp` are forever-valid bearer tokens; tokens with mismatched `iss` come from the wrong identity provider; tokens with mismatched `aud` were minted for a different relying party.
- Never reuse a JWT as a long-lived API key. JWTs are designed for short-lived authorization (minutes to hours). If you need long-lived credentials, issue an opaque API key with server-side revocation; do not extend `exp` for convenience.
- Never log full JWT tokens at info level. The token grants the bearer the rights it carries; logs are a credential exfiltration surface. Log the `sub` claim and the first 8 chars of the signature for correlation; redact the rest.
- Never use a passphrase or short string as the HS256 secret. RFC 8725 §3.5 requires entropy ≥ digest size — 256 bits (32 bytes) for HS256. Use `secrets.token_bytes(32)` at issuance and store as raw bytes.
- Never hard-code RSA public keys in source for production. JWKS endpoints exist so issuers can rotate keys without coordinated deploys; pinning the public key to a source file means rotation requires a deploy, which means rotations get postponed, which means key compromise stays exploitable longer. Use `PyJWKClient`.
- Never enumerate failure causes to unauthenticated clients. "Token expired" is a different attacker signal than "invalid signature"; respond with a uniform 401 body and put the detail in operator logs.
```

## Why this skill exists

FastAPI gives you two complementary mechanisms for wrapping requests: ASGI middleware (per-application, runs for every request before any route matches) and dependency injection (per-route, runs after route matching with full type-aware parameter resolution). FastAPI imports come from `fastapi`; ASGI middleware machinery from `starlette.middleware`; built-in middleware classes from `starlette.middleware.<name>`. Required installs: `fastapi>=0.115`, optionally `python-multipart` for form data and `uvicorn[standard]` for the dev server.

from contextlib import asynccontextmanager

from starlette.middleware import Middleware

## Class-based ASGI middleware via Middleware()

Class-based middleware is the right shape when you need per-instance configuration, lazy resource access, or interoperability with non-FastAPI ASGI apps. Pass it to `FastAPI(middleware=[Middleware(MyClass, **kwargs)])`; the kwargs are stored and applied at instantiation time when the app starts (Starlette docs, "Using middleware", verified 2026-05-01).

```python
from starlette.types import ASGIApp, Receive, Scope, Send

class CorrelationIdMiddleware:
    def __init__(self, app: ASGIApp, header_name: str = "x-correlation-id") -> None:
        self.app = app
        self.header_name = header_name

    async def __call__(self, scope: Scope, receive: Receive, send: Send) -> None:
        if scope["type"] != "http":
            await self.app(scope, receive, send)
            return
        # forward as-is; instrument send() to inject the header on the response
        async def send_wrapper(message):
            if message["type"] == "http.response.start":
                headers = list(message.get("headers", []))
                headers.append((self.header_name.encode(), b"x-corr-stub"))
                message["headers"] = headers
            await send(message)
        await self.app(scope, receive, send_wrapper)
```

Built-in classes worth defaulting on: `TrustedHostMiddleware` (rejects requests with a Host header outside an allowlist; cheap defense against host-header injection), `HTTPSRedirectMiddleware` (308-redirects bare http to https; safe behind a TLS-terminating proxy that sets `X-Forwarded-Proto`), `CORSMiddleware` (browser cross-origin policy — see guardrails), and `GZipMiddleware` (response compression for text payloads).

## Dependencies: per-route auth and validation via Depends()

Dependencies are functions that FastAPI calls before your route function, with their parameters resolved the same way route parameters are resolved. They are the right place for authentication, authorization checks, request-scoped resources (a DB connection from the pool), and shared parameter validation. Per FastAPI docs ("Dependencies", verified 2026-05-01), dependencies run AFTER middleware and AFTER route matching but BEFORE the route function body.

```python
bearer_scheme = HTTPBearer(auto_error=False)

async def current_user(
    request: Request,
    credentials: HTTPAuthorizationCredentials | None = Depends(bearer_scheme),
) -> dict:
    if credentials is None or credentials.scheme.lower() != "bearer":
        raise HTTPException(status_code=401, detail="missing bearer token")
    # delegate token verification to a dedicated function (or the JWT skill)
    claims = await verify_token(credentials.credentials)
    return claims

@app.get("/me")
async def me(user: dict = Depends(current_user)) -> dict:
    return {"sub": user["sub"]}
```

Two extension points worth knowing: sub-dependencies (a dependency can `Depends()` another, with caching scoped to one request via `use_cache=True` default), and `dependencies=[Depends(...)]` on routers and the app — useful when a check should run for every route in a group without having to add a parameter to each route function. Use the latter for "every route in this router needs an authenticated user" patterns.

Use middleware for app-wide cross-cutting concerns (CORS, host pin, HTTPS, request ID, timing). Use dependencies for per-route concerns that need typed parameter resolution (auth, role checks, request-scoped DB connections, query-parameter validation).
```

## End-to-end example: middleware stack + auth dependency, three request paths

The case-space coverage: a request that passes the middleware stack and the auth dependency (200), a request that fails CORS preflight (403/blocked by browser), and a request with no/invalid bearer token (401). One FastAPI app, all three paths.

```python
import os
from contextlib import asynccontextmanager
from fastapi import Depends, FastAPI, HTTPException, Request
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from starlette.middleware import Middleware
from starlette.middleware.cors import CORSMiddleware
from starlette.middleware.trustedhost import TrustedHostMiddleware

ALLOWED_HOSTS = ["api.example.com"]
ALLOWED_ORIGINS = ["https://app.example.com"]
bearer_scheme = HTTPBearer(auto_error=False)

async def current_user(
    credentials: HTTPAuthorizationCredentials | None = Depends(bearer_scheme),
) -> dict:
    # PATH C: no/invalid bearer → 401 from the dependency, route never runs
    if credentials is None or credentials.scheme.lower() != "bearer":
        raise HTTPException(status_code=401, detail="missing bearer token")
    claims = await verify_token(credentials.credentials)   # raises on bad token
    return claims

@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup: open shared resources here
    yield
    # shutdown: close shared resources here

middleware = [
    Middleware(TrustedHostMiddleware, allowed_hosts=ALLOWED_HOSTS),
    Middleware(
        CORSMiddleware,
        allow_origins=ALLOWED_ORIGINS,
        allow_credentials=True,
        allow_methods=["GET", "POST"],
        allow_headers=["Authorization", "Content-Type"],
    ),
]
app = FastAPI(lifespan=lifespan, middleware=middleware)

@app.get("/me")
async def me(user: dict = Depends(current_user)) -> dict:
    # PATH A: valid token → 200 with user payload
    return {"sub": user["sub"]}
```

Three paths in one app: PATH A is the happy case where TrustedHost passes, CORS preflight (if any) is permitted, and the bearer token validates; PATH B is a CORS-failing browser request (the CORSMiddleware response signals the browser to block, observable as a CORS error in dev tools); PATH C is the 401 from the auth dependency, never reaching the route function. The `lifespan` context is empty here for clarity but is where DB pools and JWKS clients open at startup and close at shutdown.

## Verification checklist

Mechanical checks for any FastAPI app's middleware and dependency stack:

- [ ] `CORSMiddleware` is configured with an explicit `allow_origins` list, never `["*"]` together with `allow_credentials=True`. `grep -rn 'allow_origins' src/` returns the configured list; if `["*"]` and `allow_credentials=True` appear in the same `Middleware(CORSMiddleware, ...)` call, that combination is invalid per the CORS spec and Starlette rejects it.
- [ ] `TrustedHostMiddleware` (or equivalent host pinning) is present and the allowlist is non-empty. `grep -rn 'TrustedHostMiddleware\|allowed_hosts' src/` returns the configuration.
- [ ] HTTPS is enforced either by `HTTPSRedirectMiddleware` or by the upstream proxy with `forwarded_allow_ips` set on uvicorn. `grep -rn 'HTTPSRedirectMiddleware\|forwarded_allow_ips' src/` returns at least one hit.
- [ ] No raw secrets (API keys, JWT secrets, DB credentials) are read inside route functions or dependencies on every call. They should be read once at lifespan startup or module import. `grep -rnE "os\.environ\[" src/ | grep -E "depends|dependency|route|@app\." returns nothing` (heuristic).
- [ ] Authentication is implemented as a dependency, not as middleware. `grep -rn '@app.middleware' src/` does not return code that calls JWT decoders, OAuth introspection, or password verification — those belong in `Depends()` chains.
- [ ] Long-lived resources (DB pools, HTTP clients, JWKS caches) open in `lifespan` and close in the same context. `grep -rn 'lifespan\|on_event' src/` — `on_event` is deprecated in FastAPI 0.110+; existing usage should migrate.
- [ ] Custom middleware preserves the response on exception paths. Inspect each `@app.middleware` and class-based middleware: `await call_next(request)` must be inside a `try` that re-raises rather than swallows, OR the middleware must construct a fallback response that is itself an ASGI-valid Response.
- [ ] `APIRoute` subclasses that capture raw bytes pair with explicit verification (e.g. webhook signature verify) before any JSON parsing dependency runs. Cross-reference with `webhook-patterns` skill for the contract.
- [ ] Middleware ordering is documented or obvious from the `middleware=[...]` list. The list ordering is the request-path order top-to-bottom; reviewers must be able to read it that way.

## Guardrails: never do these

These are the lines you do not cross when wiring FastAPI middleware and dependencies. Each item below has bitten production systems; the execution and example sections show the right shape.

- Never use `allow_origins=["*"]` together with `allow_credentials=True`. The CORS spec forbids the combination — browsers will reject the response — and Starlette refuses to start. Always enumerate origins explicitly.
- Never put authentication logic in middleware. Middleware runs before route matching, has no typed access to path/query parameters, and cannot easily express per-route policy. Use a `Depends()` chain so each route declares its auth requirement and FastAPI's dependency cache deduplicates across sub-dependencies.
- Never read secrets from `os.environ` inside a hot-path route or dependency. Read once at lifespan startup, store on `app.state`, and reference from there. Per-request env reads cost syscalls and miss any rotation that requires explicit reload.
- Never use `@app.on_event("startup")` / `@app.on_event("shutdown")` in new code. Both are deprecated as of FastAPI 0.110 (verified 2026-05-01); the `lifespan` async context manager is the supported replacement and gives you a single scope for both phases.
- Never assume middleware execution order matches the order you added them with `@app.middleware`. The decorator adds to the front of the stack, so the LAST decorator-added middleware is the FIRST to run. For predictable ordering, declare the full stack in `FastAPI(middleware=[...])` at construction time.
- Never swallow exceptions in custom middleware. Catch only what you can construct a valid response for; re-raise everything else so `ServerErrorMiddleware` can render a 500 and the exception is logged. Silencing errors in middleware is how you ship "200 OK with empty body" outage modes.
- Never call `await request.body()` or `await request.json()` in middleware unless the middleware is the only consumer of the body. Once the body is read, downstream code that expects to read it again gets an empty stream. If the middleware needs the body for inspection, use the `APIRoute` custom request pattern so the body is cached on the request object.
- Never trust the `X-Forwarded-Proto` or `X-Forwarded-For` header without configuring uvicorn's `forwarded_allow_ips` to the upstream proxy's IP. Otherwise any client can claim to be HTTPS or any IP, defeating downstream IP allowlists and HTTPS detection.
- Never mix `dependencies=[Depends(...)]` on a router with no-op route signatures expecting the same dependency injected. Pick one — the router-level `dependencies` form is for "run this for every route, no parameter access," while the per-route `Depends()` form is for "the route needs this value." Mixing them runs the dependency twice with confusing caching behavior.
- Never deploy a FastAPI app to production without `TrustedHostMiddleware`. A missing host check leaves the app responding to any Host header, which enables host-header injection attacks and cache poisoning when behind a CDN.
```

## Why this skill exists

Python async patterns center on three procedures: structured concurrency (running multiple coroutines with cleanly bounded lifetimes via `asyncio.TaskGroup`), explicit cancellation handling (treating `CancelledError` as a control-flow signal, not a generic error), and disciplined bridging to blocking work (never running CPU- or syscall-blocking code on the event loop directly; always offload via `to_thread` or `run_in_executor`). Skip any of the three and the symptom is the same — async code that "works" until it deadlocks, leaks tasks, or silently swallows partial failures.

This skill targets Python 3.11+ because `asyncio.TaskGroup`, `asyncio.timeout`, `ExceptionGroup`, and `except*` all stabilized in 3.11 (PEP 654, verified 2026-05-01 against peps.python.org and docs.python.org/3). Pre-3.11 patterns (raw `gather` with `return_exceptions=True`, manual cancellation chaining) still work but require more discipline; this skill teaches the modern surface.

Use this skill when writing new asyncio code (FastAPI handlers, background workers, fan-out fetchers); reviewing existing async code for cancellation correctness or blocking-call leaks; or migrating a pre-3.11 codebase from `asyncio.gather` to `TaskGroup`.

The skill covers the language-level async primitives. Framework integrations (FastAPI, anyio, trio compatibility shims) live in their own skills; this skill teaches what every async Python author must know regardless of framework.
```

## Setup: entry point, imports, Python version

asyncio code starts at one entry point — `asyncio.run(coro)` — which creates the event loop, runs the coroutine, finalizes async generators, and closes the loop on exit. Per docs.python.org/3/library/asyncio-runner.html (verified 2026-05-01), `asyncio.run` cannot be called when another event loop is already running in the same thread, and should be called exactly once at program top level.

```python
import asyncio
from asyncio import TaskGroup, timeout, CancelledError, to_thread
from contextlib import asynccontextmanager
```

Python version requirement: 3.11 or later for `TaskGroup`, `timeout`, and `ExceptionGroup` syntax. Check with `python --version` before authoring against this skill; pre-3.11 code paths need different shapes (see compatibility note below).

Inside any coroutine, prefer `asyncio.get_running_loop()` over `asyncio.get_event_loop()` (the latter is deprecated for coroutine use as of 3.10 and emits a `DeprecationWarning` when no loop is running, per docs.python.org/3/library/asyncio-eventloop.html, verified 2026-05-01). The running-loop function raises `RuntimeError` if called outside a running loop, which surfaces accidental sync-context calls immediately rather than silently allocating a fresh loop.

Compatibility note for pre-3.11 codebases: `TaskGroup` does not exist; use `asyncio.gather(*coros, return_exceptions=True)` and inspect each result for exception instances. `asyncio.timeout` does not exist; use `asyncio.wait_for(coro, timeout=N)` which raises `TimeoutError` on expiry. `ExceptionGroup` does not exist; collect exceptions into a list and raise the first or wrap manually.

## Structured concurrency with TaskGroup

`asyncio.TaskGroup` (Python 3.11+, verified 2026-05-01 against docs.python.org/3/library/asyncio-task.html) is the modern replacement for `asyncio.gather` when you need bounded concurrency with predictable error behavior. The `async with TaskGroup() as tg:` block creates tasks via `tg.create_task(coro)`; the block does not exit until every task either completes or raises. If any task raises, all sibling tasks are cancelled, and the group raises an `ExceptionGroup` containing every non-`CancelledError` exception that occurred.

```python
async def fetch_all(urls: list[str]) -> list[bytes]:
    async with asyncio.TaskGroup() as tg:
        tasks = [tg.create_task(fetch(url)) for url in urls]
    # all tasks are done here — completed OR raised; the group enforces it
    return [t.result() for t in tasks]
```

This pattern is structured concurrency: tasks created in the group cannot outlive the `async with` block. They cannot leak as fire-and-forget background work, they cannot escape into a parent scope. Compared to `gather(*, return_exceptions=True)` — where you must remember to inspect every result and raise manually — the TaskGroup default is correct: one task failing fails the group, the rest are cancelled, and the caller sees an `ExceptionGroup` they can dispatch on with `except*`.

The `except*` syntax (PEP 654, verified 2026-05-01) lets you handle subsets of an `ExceptionGroup` independently. Each `except*` clause matches by exception type and runs against the matching subgroup; unmatched exceptions re-raise as a residual group. `CancelledError` is excluded from `ExceptionGroup` automatically by `TaskGroup` — when the group cancels its members, those cancellations are bookkeeping, not failure signal.

## Guardrails: never do these

These are the lines you do not cross when writing asyncio code. Each item below has bitten production systems; the execution and example sections show the right shape.

- Never call a blocking function directly inside a coroutine. `requests.get`, `time.sleep`, `psycopg2.execute`, sync file IO, NumPy multi-second math — all freeze the event loop for their duration, which means every other coroutine in flight stops making progress. Wrap with `asyncio.to_thread(...)` (or use an async-native library).
- Never write `except Exception:` without re-raising `CancelledError`. `CancelledError` inherits from `BaseException` since Python 3.8 specifically so a typical `except Exception` does not catch it — but `except:` (bare) does, and `except BaseException:` does. Suppressing cancellation means a cancelled task keeps running past the cancel point.
- Never call `asyncio.gather` and ignore the returned list. With `return_exceptions=False` (default) the first exception cancels siblings and propagates; with `return_exceptions=True` the result list contains exception instances mixed with values, and discarding it silently swallows every error. Prefer `TaskGroup` for new code.
- Never `asyncio.create_task(coro)` without keeping a reference. The event loop holds a weak reference to the task; if the only reference is the local you discarded, garbage collection can finalize the task at any point — usually mid-network-call. Either store the task or use `TaskGroup` which holds references for its members.
- Never call `asyncio.run` inside a coroutine. `asyncio.run` creates a new loop; if one is already running, it raises `RuntimeError`. The error message blames the caller; the cause is "called from a coroutine when one was already running." Inside a coroutine, just `await` the inner coroutine directly.
- Never use `asyncio.get_event_loop()` inside a coroutine. It is deprecated for coroutine use as of 3.10 and removed in 3.16. Use `asyncio.get_running_loop()` which raises if there is no running loop, surfacing the bug rather than silently allocating a fresh loop.
- Never run `asyncio.run(main())` more than once in a process. The first call creates a loop, runs to completion, and closes it. The second call attempts to run on a closed loop and raises. If you need a long-lived event loop, build the loop yourself with `asyncio.new_event_loop()` and manage it explicitly — but `asyncio.run` is correct for the typical case.
- Never use `time.sleep` in async code. It blocks the event loop. `await asyncio.sleep(seconds)` yields cooperatively. The two are not interchangeable.
- Never assume `asyncio.shield(coro)` makes a task safe forever. Shield prevents outer cancellation from propagating to the inner await, but the inner task still runs on the loop and still needs a bounded lifetime (its own timeout, an external deadline). Without bounded lifetime, shielded tasks become the fire-and-forget leaks structured concurrency was designed to prevent.
- Never share a single `aiohttp.ClientSession` or `httpx.AsyncClient` across event loops. They bind to the loop they were created on; passing them to a different loop (e.g., reusing a module-level instance across `asyncio.run` calls) raises confusing runtime errors. Construct them inside the coroutine that uses them, or attach them to `app.state` in a FastAPI lifespan.

from fastapi import Depends, FastAPI, HTTPException, Request
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from starlette.middleware import Middleware
from starlette.middleware.cors import CORSMiddleware
from starlette.middleware.trustedhost import TrustedHostMiddleware

ALLOWED_HOSTS = ["api.example.com"]
ALLOWED_ORIGINS = ["https://app.example.com"]

# Import jwt after the app is defined to avoid circular imports in some setups
import jwt
from jwt.exceptions import ExpiredSignatureError, InvalidSignatureError, InvalidTokenError


async def current_user(
    credentials: HTTPAuthorizationCredentials | None = Depends(HTTPBearer(auto_error=False)),
) -> dict:
    if credentials is None or credentials.scheme.lower() != "bearer":
        raise HTTPException(status_code=401, detail="missing bearer token")
    try:
        claims = jwt.decode(
            credentials.credentials,
            key=os.environ["JWT_HS256_SECRET"].encode("utf-8"),

            issuer=os.environ["JWT_EXPECTED_ISSUER"],
            audience=os.environ["JWT_EXPECTED_AUDIENCE"],
            leeway=30,
        )
    except (ExpiredSignatureError, InvalidSignatureError, InvalidTokenError):
        raise HTTPException(status_code=401, detail="invalid token")
    return claims
