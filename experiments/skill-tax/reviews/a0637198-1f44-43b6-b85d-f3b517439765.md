## Class-based ASGI middleware via Middleware()

Class-based middleware is the right shape when you need per-instance configuration, lazy resource access, or interoperability with non-FastAPI ASGI apps. Pass it to `FastAPI(middleware=[Middleware(MyClass, **kwargs)])`; the kwargs are stored and applied at instantiation time when the app starts (Starlette docs, "Using middleware", verified 2026-05-01).

```python
from starlette.types import ASGIApp, Receive, Scope, Send

class CorrelationIdMiddleware:
    def __init__(self, app: ASGIApp, header_name: str = "x-correlation-id") -> None:
        self.app = app
        self.header_name = header_name

    async def __call__(self, scope: Scope, receive: Receive, send: Send) -> None:
        if scope["type"] != "http":
            await self.app(scope, receive, send)
            return
        # forward as-is; instrument send() to inject the header on the response
        async def send_wrapper(message):
            if message["type"] == "http.response.start":
                headers = list(message.get("headers", []))
                headers.append((self.header_name.encode(), b"x-corr-stub"))
                message["headers"] = headers
            await send(message)
        await self.app(scope, receive, send_wrapper)
```

Built-in classes worth defaulting on: `TrustedHostMiddleware` (rejects requests with a Host header outside an allowlist; cheap defense against host-header injection), `HTTPSRedirectMiddleware` (308-redirects bare http to https; safe behind a TLS-terminating proxy that sets `X-Forwarded-Proto`), `CORSMiddleware` (browser cross-origin policy ‚Äî see guardrails), and `GZipMiddleware` (response compression for text payloads).

## Dependencies: per-route auth and validation via Depends()

Dependencies are functions that FastAPI calls before your route function, with their parameters resolved the same way route parameters are resolved. They are the right place for authentication, authorization checks, request-scoped resources (a DB connection from the pool), and shared parameter validation. Per FastAPI docs ("Dependencies", verified 2026-05-01), dependencies run AFTER middleware and AFTER route matching but BEFORE the route function body.

```python
bearer_scheme = HTTPBearer(auto_error=False)

async def current_user(
    request: Request,
    credentials: HTTPAuthorizationCredentials | None = Depends(bearer_scheme),
) -> dict:
    if credentials is None or credentials.scheme.lower() != "bearer":
        raise HTTPException(status_code=401, detail="missing bearer token")
    # delegate token verification to a dedicated function (or the JWT skill)
    claims = await verify_token(credentials.credentials)
    return claims

@app.get("/me")
async def me(user: dict = Depends(current_user)) -> dict:
    return {"sub": user["sub"]}
```

Two extension points worth knowing: sub-dependencies (a dependency can `Depends()` another, with caching scoped to one request via `use_cache=True` default), and `dependencies=[Depends(...)]` on routers and the app ‚Äî useful when a check should run for every route in a group without having to add a parameter to each route function. Use the latter for "every route in this router needs an authenticated user" patterns.

Use middleware for app-wide cross-cutting concerns (CORS, host pin, HTTPS, request ID, timing). Use dependencies for per-route concerns that need typed parameter resolution (auth, role checks, request-scoped DB connections, query-parameter validation).

## Setup: entry point, imports, Python version

asyncio code starts at one entry point ‚Äî `asyncio.run(coro)` ‚Äî which creates the event loop, runs the coroutine, finalizes async generators, and closes the loop on exit. Per docs.python.org/3/library/asyncio-runner.html (verified 2026-05-01), `asyncio.run` cannot be called when another event loop is already running in the same thread, and should be called exactly once at program top level.

```python
import asyncio
from asyncio import TaskGroup, timeout, CancelledError, to_thread
from contextlib import asynccontextmanager
```

Python version requirement: 3.11 or later for `TaskGroup`, `timeout`, and `ExceptionGroup` syntax. Check with `python --version` before authoring against this skill; pre-3.11 code paths need different shapes (see compatibility note below).

Inside any coroutine, prefer `asyncio.get_running_loop()` over `asyncio.get_event_loop()` (the latter is deprecated for coroutine use as of 3.10 and emits a `DeprecationWarning` when no loop is running, per docs.python.org/3/library/asyncio-eventloop.html, verified 2026-05-01). The running-loop function raises `RuntimeError` if called outside a running loop, which surfaces accidental sync-context calls immediately rather than silently allocating a fresh loop.

Compatibility note for pre-3.11 codebases: `TaskGroup` does not exist; use `asyncio.gather(*coros, return_exceptions=True)` and inspect each result for exception instances. `asyncio.timeout` does not exist; use `asyncio.wait_for(coro, timeout=N)` which raises `TimeoutError` on expiry. `ExceptionGroup` does not exist; collect exceptions into a list and raise the first or wrap manually.

## Structured concurrency with TaskGroup

`asyncio.TaskGroup` (Python 3.11+, verified 2026-05-01 against docs.python.org/3/library/asyncio-task.html) is the modern replacement for `asyncio.gather` when you need bounded concurrency with predictable error behavior. The `async with TaskGroup() as tg:` block creates tasks via `tg.create_task(coro)`; the block does not exit until every task either completes or raises. If any task raises, all sibling tasks are cancelled, and the group raises an `ExceptionGroup` containing every non-`CancelledError` exception that occurred.

```python
async def fetch_all(urls: list[str]) -> list[bytes]:
    async with asyncio.TaskGroup() as tg:
        tasks = [tg.create_task(fetch(url)) for url in urls]
    # all tasks are done here ‚Äî completed OR raised; the group enforces it
    return [t.result() for t in tasks]
```

This pattern is structured concurrency: tasks created in the group cannot outlive the `async with` block. They cannot leak as fire-and-forget background work, they cannot escape into a parent scope. Compared to `gather(*, return_exceptions=True)` ‚Äî where you must remember to inspect every result and raise manually ‚Äî the TaskGroup default is correct: one task failing fails the group, the rest are cancelled, and the caller sees an `ExceptionGroup` they can dispatch on with `except*`.

The `except*` syntax (PEP 654, verified 2026-05-01) lets you handle subsets of an `ExceptionGroup` independently. Each `except*` clause matches by exception type and runs against the matching subgroup; unmatched exceptions re-raise as a residual group. `CancelledError` is excluded from `ExceptionGroup` automatically by `TaskGroup` ‚Äî when the group cancels its members, those cancellations are bookkeeping, not failure signal.

## Bridging sync and async: to_thread and run_in_executor

Blocking calls ‚Äî `requests.get`, `psycopg2.execute`, `time.sleep`, file IO, CPU-bound math ‚Äî run on the event loop thread by default and freeze the loop for their duration. The fix is to offload to a thread pool. Per docs.python.org/3/library/asyncio-eventloop.html (verified 2026-05-01), the modern shape is `asyncio.to_thread(func, *args)`, added in 3.9, which uses the default thread executor and returns an awaitable result. For non-default executors (a process pool for CPU-bound work, a custom-sized thread pool for high-concurrency IO), use `loop.run_in_executor(executor, func, *args)`.

```python
import asyncio
import time

def cpu_heavy(n: int) -> int:
    return sum(i * i for i in range(n))

async def safe_call() -> int:
    # offload ‚Äî event loop stays responsive for other tasks
    return await asyncio.to_thread(cpu_heavy, 10_000_000)

async def with_process_pool() -> int:
    loop = asyncio.get_running_loop()
    with concurrent.futures.ProcessPoolExecutor() as pool:
        return await loop.run_in_executor(pool, cpu_heavy, 10_000_000)
```

Rule of thumb: any function that blocks for more than a few milliseconds ‚Äî file IO, network IO via a sync library, CPU work, database calls via a sync driver ‚Äî must be wrapped. The cost of forgetting is one slow request taking down the entire event loop's concurrency budget.
